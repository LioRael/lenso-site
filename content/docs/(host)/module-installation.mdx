title: Module Installation
description: Install linked modules, service-backed module releases, catalog entries, console bundles, and service plans.
---

Installing a module is a host-local operator action. Discovery and installation
are separate on purpose.

For the discovery side of this split, read
[Module Catalogs](/docs/module-catalogs).

## Linked first-party modules

First-party linked modules are installed by name:

```sh
lenso module install auth
lenso module install auth-password
lenso module install auth-device
```

`auth-password` and `auth-device` depend on `auth`. Runtime Console exposes
declared dependencies through module metadata so the operator can see why a
module is not ready.

Profiles can change install effects:

```sh
lenso module install auth --profile redis-session-cache
```

The Redis session profile enables the auth Redis feature, writes `REDIS_URL`,
and records `auth.session_cache=redis` as a runtime default.

## Service-backed modules

Install a packaged business module through its module release artifact:

```sh
lenso module release inspect dist/lenso-service/support-suite-provider/modules/support-ticket/lenso.module-release.json
lenso module install dist/lenso-service/support-suite-provider/modules/support-ticket/lenso.module-release.json \
  --base-url http://127.0.0.1:4110/lenso/service/v1
```

The module release points at the provider service package or service manifest.
The host still owns install state, queues, retries, Runtime Stories, and Console
evidence.

Legacy remote module manifests remain installable for compatibility:

```sh
lenso module install http://127.0.0.1:4110/lenso/module/v1/manifest
lenso module install ./lenso.module.json --base-url https://example.com/lenso/module/v1
```

After installation, restart the API and worker so the host reloads configured
remote sources and console extension registry state.

## What install writes

The service/module install lane writes host-local files:

| File | Purpose |
| --- | --- |
| `.env` | updates `REMOTE_MODULES=<provider>=<base_url>` and declared env values |
| `.lenso/module-installs.json` | install receipt used by uninstall |
| `.lenso/module-services.json` | service commands declared by the module |
| `.lenso/console/extensions/<module>/` | copied same-origin console bundles |
| `.lenso/console/extensions/registry.json` | hosted Console extension registry |

Install commands from a manifest are planned but not run unless the operator
opts in:

```sh
lenso module install https://example.com/lenso/module/v1/manifest --run-install-commands
```

## Catalog is discovery

Add a module to the local discovery catalog:

```sh
lenso module catalog add dist/lenso-service/support-suite-provider/modules/support-ticket/lenso.module-release.json \
  --base-url http://127.0.0.1:4110/lenso/service/v1
```

This writes `.lenso/module-catalog.json`. It does not install the module.

Runtime Console's Available Modules view reads, in order:

1. local `.lenso/module-catalog.json`
2. already configured service or remote sources
3. built-in `builtin:lenso-official-module-catalog`

The built-in catalog is only a fallback discovery source. It is not a hidden
install database.

## Runtime Console install flow

Runtime Console can write the same host-local install state through:

```text
POST /admin/data/available-modules/{module}/install
```

This is still host-owned installation. The browser does not install npm
packages or compile third-party code into the official Console bundle.

## Doctor and uninstall

Check installed service-backed module state:

```sh
lenso module doctor
lenso module doctor billing
```

Remove host-local remote source and console registry entries:

```sh
lenso module uninstall billing
```

Uninstall leaves module-owned data alone. It removes the host's local install
state; it does not drop remote service databases or business records.
