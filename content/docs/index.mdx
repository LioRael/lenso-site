Lenso is an agent-ready modular app framework for Rust business systems.

Start with one deployable backend, add business modules, then verify the system
through the Runtime Console. These docs are organized around the two jobs users
usually have on day one:

- Host builders create and operate the app that loads modules.
- Module authors add business capability through manifests, routes, actions,
  runtime functions, and optional console surfaces.

## Choose Your Path

- New host app: read [Host Quickstart](/docs/quickstart) and finish with the
  API, worker, Postgres, and `/console` running.
- Product-shaped app: read [Product Blueprints](/docs/product-blueprints) when
  you want a generated service-ready app plus supported addons.
- Host configuration: read [Configuration](/docs/configuration) when you need
  real env variables for profiles, CORS, Redis, telemetry, Console paths, or
  service tokens.
- Host composition: read [Host Composition](/docs/host-composition) when you
  need to understand profiles, `HostBuilder`, linked modules, service sources,
  and module startup toggles.
- Platform model: read [Platform Concepts](/docs/platform-concepts) to
  understand manifests, host-owned runtime work, and remote boundaries.
- Runtime evidence model: read [Runtime Stories](/docs/runtime-stories) to
  understand Lenso's story graph, correlation model, and operator-readable
  business evidence.
- Manifest fields: read [Manifest Reference](/docs/manifest-reference) when you
  need exact `ModuleManifest` fields, builder methods, capabilities, route
  metadata, and lint categories.
- API integrations: read [API Clients](/docs/api-clients) when consuming Lenso
  HTTP APIs from OpenAPI or handling the standard error envelope.
- Contract work: read [Contracts and Checks](/docs/contracts-and-checks) before
  changing generated artifacts, OpenAPI, gRPC, or architecture-sensitive code.
- Release safety: read [Checks and Release](/docs/checks-and-release) when you
  need to know which `just` gate matches a code, contract, Console, or release
  change.
- CLI work: read [CLI Reference](/docs/cli-reference) for `lenso host`,
  `lenso app`, `lenso serve`, `lenso module`, module release checks, catalogs,
  marketplace compatibility, and Console package plans.
- Install flow: read [Module Installation](/docs/module-installation) when you
  need to know what the CLI writes locally and how a running host registers
  modules through the admin API.
- Catalogs: read [Module Catalogs](/docs/module-catalogs) when you need the
  difference between discovery, install, and the official fallback catalog.
- Deployment: read [Deployment](/docs/deployment) before packaging a generated
  host, publishing a hosted Console bundle, or wiring migrations and workers.
- Data changes: read [Database and Migrations](/docs/database-migrations) before
  adding host-linked module SQL or running deploy migrations.
- Operations: read [Observability](/docs/observability) for health probes,
  structured logs, OTLP traces, and Runtime Console runtime views.
- Host verification: read [Runtime Console](/docs/runtime-console) and confirm
  module health, data, actions, and runtime evidence.
- Broken local setup: read [Troubleshooting](/docs/troubleshooting) to diagnose
  blank Console, missing modules, remote service readiness, CORS, Redis,
  generated artifacts, busy ports, and release-gate failures.
- Built-in auth: read [Built-in Modules](/docs/built-in-modules) and install
  `auth`, `auth-anonymous`, `auth-password`, or `auth-device` into a host.
- Rust module: read [Module Authoring](/docs/module-authoring) and scaffold one
  useful capability with a smoke check.
- Operator surfaces: read [Admin Surfaces](/docs/admin-surfaces) when the module
  needs data pages, actions, query values, or custom Console UI.
- Auth and permissions: read
  [Auth and Capabilities](/docs/auth-capabilities) when a module needs actor
  identity, session integration, or capability-scoped operations.
- Runtime work: read [Runtime and Lifecycle](/docs/runtime-lifecycle) when you
  need handlers, lifecycle declarations, background workers, or remote runtime
  endpoints.
- Console package: read [Console Packages](/docs/console-packages) when a module
  ships a typed React package for the Runtime Console.
- Out-of-process module: publish a service-backed module release, then use
  [Module Installation](/docs/module-installation) to install it into a host.
- Remote protocol: read [Remote Module Protocol](/docs/remote-protocol) when
  implementing HTTP JSON or gRPC remote modules outside the TypeScript kit.
- Agent-assisted development: read [Agent Development](/docs/agent-development)
  and use skills, scaffolds, checks, and `/console` evidence.
- Working reference: read [Examples](/docs/examples) and run the support-ticket
  or account-profile smokes.

## What to install

- Public skills: `npx skills add lenso-dev/skills`.
- `lenso@0.3.16` is the public Rust facade for module-authoring declarations and the `host` feature.
- `lenso-cli@0.1.26` scaffolds host apps with `lenso host init <dir>`.
- `@lenso/remote-module-kit@0.1.1` supports out-of-process JavaScript and TypeScript modules.
- Runtime Console is bundled into generated hosts by published CLI builds.

## Mental Model

Lenso is modular-first and service-ready, but it is not a Kubernetes-first
microservice platform. Build one deployable system first. Split selected
modules only when their ownership, data, and operational boundaries are real.
